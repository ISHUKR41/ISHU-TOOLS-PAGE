## PART 0 — FULL CODEBASE DEBUG & AUDIT (DO THIS FIRST)

Before making any improvements, the AI must perform a full audit:

### 0.1 File & Folder Debugging

- Scan every file in the repository for syntax errors, broken imports, dead code, and missing dependencies.
- Check every `package.json` or dependency file and flag outdated packages or missing packages.
- Identify all files that reference libraries or APIs that no longer work or have changed their interface.
- Find all hardcoded API keys, tokens, or secrets and flag them for environment variable migration.
- Check for duplicate files, duplicate tool definitions, and conflicting routes.
- Ensure all file names follow consistent naming conventions (no mixed cases, no spaces).
- Check all static assets (images, icons, fonts) — remove unused ones, compress oversized ones.
- Verify that all internal links and routing paths are correct and not broken.
- Check all `import`/`require` statements — remove those that import things never used.
- Scan CSS/Tailwind classes for conflicts, overrides, and unused declarations.
- Check if the PWA manifest, service worker, and offline cache are functioning correctly.
- Verify the sitemap.xml and robots.txt are up to date with all tool routes.
- Check all meta tags (OG, Twitter Card, canonical URLs) per tool page.
- Confirm all tool pages have proper `<title>`, `<description>`, and structured data (JSON-LD).
- Look for any memory leaks — e.g., event listeners not removed after component unmounts, intervals not cleared.
- Identify tools that currently crash or return blank output and mark them as Priority 1 fixes.
- Check all server-side endpoints for timeout handling, CORS issues, and missing error responses.
- Verify file upload logic — max file size enforcement, file type validation, and cleanup after processing.
- Confirm all download triggers work across Chrome, Firefox, Safari, and mobile browsers.

### 0.2 Performance Audit

- Measure and flag any tool page taking more than 2 seconds to load.
- Check for unnecessary large JavaScript bundles being loaded on pages that don't need them.
- Lazy-load heavy libraries (FFmpeg, Tesseract, PDF.js, etc.) — they should not block page load.
- Check for render-blocking scripts and stylesheets.
- Ensure images on the homepage and tool pages use modern formats (WebP) and lazy loading.
- Confirm fonts are loaded with `font-display: swap` or preloaded.
- Verify that search indexing of 1299 tools is fast — use a pre-built index (e.g., Fuse.js or FlexSearch), not real-time DOM scraping.

## PART 4 — VIDEO & MEDIA TOOLS — DETAILED IMPROVEMENT

### 4.1 General Rules for ALL Video Tools

This is the most error-prone category. The following fallback architecture MUST be implemented:

1. **Primary:** FFmpeg.wasm (client-side WebAssembly — works in browser, no upload needed for small files)
2. **Fallback 1:** Server-side FFmpeg via API endpoint (for files where WASM runs out of memory)
3. **Fallback 2:** yt-dlp (server-side, for download tools only)
4. **Fallback 3:** Alternative library (e.g., fluent-ffmpeg on Node.js)
5. **Fallback 4:** A second yt-dlp instance with different options/cookies
6. **Fallback 5:** Clear error message explaining what happened + "Try again later" + error log

Every video tool must:
- Show a real-time progress bar (FFmpeg provides % progress via its stderr output — parse and display it)
- Detect file size before processing — warn if >500MB
- Show estimated processing time before starting
- Never time out silently — always show a spinner and "Still processing..." after 10 seconds
- Offer a cancel button that actually stops the processing
- After completion: show video duration, resolution, and file size of output
- Download link must be a direct file download, not a redirect that breaks on mobile

### 4.2 Video Downloader (YouTube, Instagram, TikTok, Facebook, Twitter/X, etc.)

This is the tool users complain about most (as mentioned). The fix:

- Fallback 4: For YouTube specifically — ytdl-core (Node.js).

- Show video thumbnail and title after URL is pasted (before download starts).
- Add format selector: MP4, WebM, MP3 (audio only), M4A (audio only).
- Rate limiting handling: If yt-dlp returns a rate-limit error, wait 2 seconds and retry with a different fallback.

## PART 15 — SECURITY & STABILITY

### 15.1 File Upload Security

- Validate file type by magic bytes (not just extension) for ALL file-upload tools.
- Enforce strict file size limits per tool category.
- Scan uploaded files for embedded scripts (PDFs especially can contain JavaScript).
- Delete uploaded files from server within 60 seconds of processing completion.
- Never expose internal server paths in error messages or response headers.

### 15.2 Rate Limiting

- Implement rate limiting on all server-side API endpoints: max 30 requests per minute per IP.
- For download tools (video downloader especially): max 5 requests per minute per IP.
- Show a friendly rate-limit message with countdown timer if limit is hit.

### 15.3 Privacy

- Add a clear notice on file-upload tools: "Your file is processed on our server and deleted within 60 seconds."
- Add a Privacy Policy page that explains data handling.
- No tracking of individual file contents — only anonymised usage metrics.

### 15.4 Error Handling (Global)

- All API endpoints must return proper HTTP status codes (not just 200 for everything).
- All endpoints must return JSON error responses: `{ "error": true, "message": "Human-readable message", "code": "ERROR_CODE" }`.
- Client-side must catch ALL unhandled Promise rejections and display a friendly error.
- Never show a blank white page — always show a "Something went wrong. Please try again." message.
- Add a global error boundary that catches React/JS errors and shows a retry button.

- The tool directory list must be built at compile time (static), not fetched on every page load.

### 16.2 CDN & Loading

- All heavy libraries (FFmpeg.wasm, Tesseract.js, face-api.js) must be loaded from a CDN with the smallest possible initial bundle.
- Use code splitting — each tool page loads only the libraries it needs.
- Preload the most popular tools (Merge PDF, Compress Image, Remove Background) for faster cold start.

### 16.3 PWA Improvements

- Service worker must pre-cache the home page and top 10 most visited tools.
- Offline mode: Show cached version of any previously visited tool page.
- App install prompt: Show "Install ISHU TOOLS" prompt after user has visited 3+ times.

### 16.4 Analytics (Privacy-Respecting)

- Use Plausible or Umami (open-source, no cookie, GDPR-compliant) instead of Google Analytics if switching is possible.
- Track: page views, tool usage counts, error rates per tool, download counts.
- Use this data to identify the top failing tools and prioritise fixes.

## PART 17 — MOBILE-SPECIFIC IMPROVEMENTS

- All file upload inputs must trigger the native camera option on mobile (add `capture="environment"` where relevant, e.g., Scan to PDF, Remove Background from photo).
- All sliders must be touch-friendly (minimum 40px track height on mobile).
- All text areas must not zoom in on iOS (font-size minimum 16px).
- Long processing tools must keep the screen awake using Wake Lock API.
- Share button on result pages must use Web Share API on mobile.
- All video/audio previews must use the native `<video>` and `<audio>` elements (not custom players that fail on iOS).
- The tool directory must be infinite-scroll or paginated — loading 1299 tools at once is too heavy for mobile.
