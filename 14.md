Before touching any feature or UI, do a complete audit of every single file in the repository. This is mandatory.

**What to check in every file:**
- All imports — check if every imported module, component, utility, or library actually exists and is correctly spelled
- All exports — check if every exported function, component, or constant is actually used somewhere
- All environment variables — check if every `process.env.XYZ` or similar reference has a corresponding value defined
- All API calls — check every fetch/axios/HTTP call and verify: URL is correct, method is correct, headers are correct, error handling exists
- All async/await blocks — every async function must have a proper try-catch or .catch() handler
- All file paths — relative imports like `../../utils/helper` must actually point to existing files
- All hardcoded values — no hardcoded API keys, secrets, or credentials anywhere
- All console.log, console.error statements — remove all debug logs from production code
- All TODO/FIXME comments — address every single one
- Dead code removal — remove any function, variable, or component that is defined but never used
- Circular dependency check — identify and fix any circular imports
- Unused dependencies in package.json — remove packages that are installed but never used
- Missing dependencies — identify packages used in code but not listed in package.json

**Folder-level audit:**
- `/pages` or `/app` folder — every route file must render correctly, no broken routes
- `/components` folder — every component must have proper props validation and default values
- `/utils` or `/lib` folder — every utility function must handle edge cases (null, undefined, empty string, wrong type)
- `/api` folder — every API route must have proper error handling, status codes, and response formats
- `/public` folder — check all static assets (images, icons, fonts) — remove unused ones, fix broken references
- `/styles` or `/css` folder — check all CSS/SCSS files for unused rules, broken variables, conflicting styles
- `/hooks` folder (if exists) — every custom hook must handle loading, error, and success states
- `/context` folder (if exists) — every context provider must handle null states safely
- `/config` folder — all configuration files must be consistent across environments (dev, staging, prod)

**Specific bug patterns to find and fix across ALL files:**
- `undefined is not a function` — add null checks before calling methods
- `Cannot read property 'x' of undefined` — add optional chaining `?.` everywhere
- `Network request failed` — wrap in try-catch with fallback
- Broken image `src` — add `onError` fallback image
- Infinite re-renders in React — fix missing dependency arrays in useEffect
- Memory leaks — cancel async operations when component unmounts
- Unhandled promise rejections — every `.then()` needs a `.catch()`
- localStorage/sessionStorage access without availability check — add try-catch since Safari private mode throws
- File reading without size validation — add file size checks before processing
- File type validation missing — add MIME type + extension double validation

## 📌 SECTION 2 — PHASE 2: GLOBAL FALLBACK ARCHITECTURE

This is the most important section. Every single tool must implement this fallback architecture.

### 2.1 — The 6-Layer Fallback Philosophy

**Core Principle:** The user should NEVER see a broken tool. If method 1 fails, method 2 kicks in automatically. If method 2 fails, method 3 activates. The user only sees a friendly loading state and then a result — never an error, never a crash.

**Layer 1 — Primary Method:** The best, fastest, most reliable approach using the best available library.  
**Layer 2 — Secondary Library:** A completely different library that does the same job.  
**Layer 3 — Tertiary Library or API:** A third completely different approach (different library, different algorithm, or different API endpoint).  
**Layer 4 — Server-Side Fallback:** If all client-side approaches fail, try a server-side/API-based approach.  
**Layer 5 — Alternative Format Fallback:** If the exact output format fails, try an alternative format that achieves the same goal for the user.  
**Layer 6 — Graceful Degradation:** If everything fails, show the user a friendly message with alternative options — never a raw error. Offer to retry, offer a different approach, or offer to email them the result.

**Fallback Transition Rules:**
- Never show the user that a fallback was triggered — transitions must be invisible
- Each fallback attempt must have a timeout (never let one method hang forever)
- Log all fallback triggers to a silent error reporting system (e.g., Sentry, LogRocket, or custom logger)
- If fallback 6 is reached, increment a counter for that tool — if a tool reaches fallback 6 more than X times per day, send an alert to the developer

### 2.2 — Fallback Timeout Strategy

For every tool operation, define maximum time limits:
- **Fast operations** (text processing, calculations): 3 second timeout per attempt, then switch fallback
- **Medium operations** (image processing, file conversion): 10 second timeout per attempt
- **Heavy operations** (PDF processing, video handling): 30 second timeout per attempt
- **Network operations** (API calls): 8 second timeout with 3 automatic retries with exponential backoff

### 2.3 — Error Message Standards

Every tool must display errors in this format (never raw errors):
- **Primary message:** Simple, friendly, Hindi-friendly English (e.g., "Something went wrong. We're trying another method...")
- **Secondary message:** What the user can do (e.g., "Try a smaller file" or "Check your internet connection")
- **Action button:** A clear button to retry or try a different approach
- **Never show:** Stack traces, library names, HTTP status codes, or technical jargon to end users

## 📌 SECTION 5 — PHASE 5: PERFORMANCE OPTIMIZATION

### 5.1 — Core Web Vitals

Every page must score:
- LCP (Largest Contentful Paint): Under 2.5 seconds
- FID (First Input Delay): Under 100ms
- CLS (Cumulative Layout Shift): Under 0.1
- FCP (First Contentful Paint): Under 1.8 seconds
- TTFB (Time to First Byte): Under 600ms

### 5.2 — Code Splitting and Lazy Loading

- Every tool page must only load the JavaScript needed for that specific tool
- Libraries for PDF processing must NOT be loaded on image tool pages
- Video processing libraries must be lazy-loaded only when user initiates a download
- Heavy libraries (FFmpeg.wasm, Tesseract.js, TensorFlow.js) must be loaded on-demand with a visible loading indicator
- All images must use lazy loading attribute
- All non-critical CSS must be deferred

### 5.3 — Caching Strategy

- Static assets (JS, CSS, images, fonts) must have long cache headers with content hashing
- Tool results that can be cached (e.g., same file compressed = same result) must be cached in IndexedDB
- Service Worker must be implemented for offline capability on simple tools
- API responses must be cached appropriately

### 5.4 — Bundle Size Optimization

- Tree-shake all libraries — only import what is used
- Use lighter alternatives where possible (e.g., day.js instead of moment.js)
- Compress all static assets (gzip/brotli)
- Use WebP for all images in the website itself
- Font subsetting for any custom fonts used

## 📌 SECTION 7 — PHASE 7: ACCESSIBILITY

Every tool must meet WCAG 2.1 Level AA accessibility standards:
- All interactive elements must be keyboard navigable
- All images must have meaningful alt text
- Color contrast ratio must be at least 4.5:1 for normal text
- All form inputs must have associated labels
- Error messages must be programmatically associated with their inputs
- Loading states must be announced to screen readers (use ARIA live regions)
- Focus management — when a modal/dialog opens, focus must move to it; when it closes, focus must return
- No flashing content that could trigger seizures
- Support for browser zoom up to 200% without loss of functionality
- Touch target size: minimum 44x44px on touch devices

## 📌 SECTION 9 — PHASE 9: SECURITY

- **Content Security Policy (CSP):** Implement strict CSP headers to prevent XSS attacks
- **File Upload Security:** Every uploaded file must be:
  - Size-limited (define max per tool type — e.g., 50MB for PDF, 20MB for images)
  - Type-validated (both MIME type AND magic bytes, not just extension)
  - Scanned for malicious content (at minimum, validate structure)
  - Processed in an isolated context (WebWorker)
  - Never stored permanently on server unless absolutely necessary
  - Automatically deleted from server after processing
- **Rate Limiting:** Implement rate limiting on all server-side APIs to prevent abuse
- **CORS:** Properly configured CORS headers on all API endpoints
- **HTTPS Only:** Enforce HTTPS, redirect all HTTP to HTTPS
- **Input Sanitization:** Sanitize ALL user inputs before processing
- **Dependency Security:** Run `npm audit` and fix all vulnerabilities; set up Dependabot or similar automated alerts
- **API Key Security:** All API keys in environment variables, never in client-side code

## 📌 SECTION 13 — SPECIFIC INSTRUCTIONS FOR THE IMPLEMENTING AI

If you are an AI reading this document to implement the improvements, follow these rules:

1. **First, carefully read and understand the complete repository structure.** Map every file and folder before writing a single line of code.

2. **For each tool, follow this process:**
   - Read existing code completely and understand what it does
   - Identify all current bugs and list them
   - Identify all missing features from this document
   - Design the improved version with all fallbacks
   - Implement changes carefully, testing each change
   - Never remove working functionality — only add to it

3. **For fallbacks:**
   - Implement each fallback as a completely independent function/method
   - Test each fallback independently to ensure it works
   - Test the fallback switching logic (manually trigger primary failure to ensure secondary activates)
   - Ensure fallback switching is invisible to the user — they should only see a brief loading state

4. **For UI changes:**
   - Never modify tool logic while making UI changes — separate concerns
   - Test the tool functionality before and after UI changes to ensure nothing broke
   - Use CSS variables for all colors and spacing to make future changes easier

5. **For debugging:**
   - Fix bugs one at a time
   - After each fix, verify the fix didn't break anything else
   - Keep a log of what was fixed (useful for the project changelog)

6. **Do NOT:**
   - Over-engineer simple tools with unnecessary complexity
   - Add features that slow down the tool for average users
   - Break existing tool URLs (these are indexed by Google and important for SEO)
   - Remove any existing SEO content on tool pages
   - Add third-party trackers or ads without owner permission
   - Store user files permanently on server

7. **Always verify:**
   - That existing tools still work after any change
   - That new features don't conflict with existing functionality
   - That all changes are mobile-compatible
   - That all changes work across the supported browsers listed above
