## PHASE 0 — FULL CODEBASE DEBUG (DO THIS FIRST, BEFORE ANYTHING ELSE)

### 0.1 — Repository Audit

Clone the repository and do a complete audit:

- Scan every file and folder. Identify dead code, unused imports, commented-out blocks, orphan files, and broken links.
- Check every API endpoint or external service call for timeout handling and error catching.
- Look for any `console.error` or `console.log` statements that expose internals to users — wrap these properly.
- Identify all hardcoded URLs, API keys, or secrets and flag them for environment variable migration.
- Check all `package.json` / dependency files — identify outdated packages, security vulnerabilities, and unnecessary dependencies.
- Verify every route defined in the router actually has a corresponding tool page.
- Check for duplicate tool entries in the directory — the website shows some tools appear to be registered multiple times under different slugs.
- Audit all CSS/style files for redundant rules, conflicting classes, and unused selectors.
- Check for memory leaks: any file upload handlers that don't clean up blobs or object URLs after use.
- Verify that all tool pages have proper `<title>`, `<meta description>`, `<og:*>` tags for SEO.
- Check mobile responsiveness on every single tool page — especially file upload areas and output displays.
- Verify that the search feature (the "/" shortcut search bar) correctly indexes all 1247 tools.
- Check keyboard accessibility — every interactive element should be reachable via Tab and operable via Enter/Space.
- Audit loading states — every tool that does processing must show a loading spinner or skeleton, never a frozen screen.
- Check that all download buttons actually trigger file downloads and don't open in a new tab unexpectedly.

### 0.2 — Common Bugs to Specifically Look For and Fix

- **Video/Audio downloader tools:** These are the most problematic. The backend or API call likely has a single point of failure. Wrap every download attempt in a try-catch-retry loop with at least 5 different extraction methods or proxy fallbacks.
- **File upload tools:** Check that files above browser memory limits are handled gracefully with a size warning, not a crash.
- **Image compression tools:** Verify the binary search quality algorithm for hitting target KB sizes — it likely hits infinite loops or overshoots on edge cases.
- **PDF tools:** Ensure Ghostscript or whatever PDF library is used has proper error handling when the input PDF is password-protected, corrupt, or in an unusual version.
- **OCR tools:** If Tesseract or similar is used, ensure it doesn't time out silently on large images.
- **Currency converter:** Verify the live exchange rate API has a fallback to a cached rate if the API call fails.
- **Text-to-Speech:** Check that the language selector actually changes the voice and doesn't silently default to English always.
- **QR Code generator:** Verify edge cases — very long URLs, special characters, emojis.
- **Background remover:** This likely uses a third-party AI API. Ensure it has a timeout, retry, and a fallback message.
- **All hash generators (MD5, SHA):** Should work entirely client-side. If they hit any async issue, check for race conditions.

## PHASE 1 — UI/UX GLOBAL IMPROVEMENTS

### 1.1 — Design Philosophy

The current design is clean and functional. The goal is NOT to redesign it. The goal is to:

- Make it feel 10–15% more polished and trustworthy.
- Reduce visual noise and improve spacing.
- Make interactive states (hover, focus, active, loading) feel snappy and satisfying.
- Keep it lightweight — no heavy animations that hurt performance on mobile.

### 1.2 — Global Changes to Apply Across ALL Pages

**Typography:**
- Ensure consistent font sizing hierarchy across all tool pages. H1 for tool name, H2 for sections, clear body text.
- Line height should be comfortable — around 1.5–1.6 for body text.
- Reduce font weight inconsistencies — some tools likely use inconsistent bold vs regular.

**Color & Contrast:**
- Audit all text-background combinations for WCAG AA compliance (minimum 4.5:1 ratio for normal text).
- The primary brand color should be consistently applied to all CTAs, active states, and focus rings.
- Ensure dark mode (if implemented or to be implemented) properly inverts all custom colors, not just system ones.

**Spacing:**
- Standardize padding inside tool cards, input areas, and output areas.
- Ensure there is always breathing room between the tool's input section and output section.
- On mobile, ensure buttons are at least 44×44px touch targets.

**Buttons:**
- All primary action buttons (Run, Convert, Compress, Download, etc.) should have consistent sizing and style.
- Add subtle loading state to every button — when clicked, the button should show a spinner and become disabled until the operation completes.
- Success state: briefly flash green or show a checkmark after successful operations.
- Error state: shake animation or red border to indicate failure, with a clear message.

**File Upload Areas:**
- All drag-and-drop zones should show clear visual feedback when a file is dragged over them.
- Show file name and size after selection, before the user clicks Run.
- For multi-file tools, show a list of all selected files with individual remove buttons.
- If the file type is wrong, show an immediate inline error — don't wait for the user to click Run.

**Output/Result Areas:**
- Output sections should appear smoothly (subtle fade or slide in) after processing completes.
- For text output, add a one-click "Copy to Clipboard" button that changes to "Copied!" for 2 seconds.
- For file downloads, show the file name, size, and format before the download button.
- If processing takes more than 2 seconds, show a progress indicator or at minimum a spinner.

**Navigation:**
- The top navigation bar (logo, quick tool links, search) should be sticky and not take up too much vertical space on mobile.
- The search bar (/) should auto-focus when "/" is pressed anywhere on the page, and close cleanly on Escape.
- Category navigation on the main directory page should be horizontally scrollable on mobile.

**Footer:**
- Footer links should all open correctly.
- Social media links (LinkedIn, Instagram, YouTube, X) should open in new tabs.
- Ensure the "Install ISHU TOOLS App" PWA prompt works correctly on Android Chrome and iOS Safari.

**Error Pages:**

## PHASE 2 — TOOL-BY-TOOL IMPROVEMENT GUIDE

For EVERY tool listed below, apply ALL of the following universal improvements FIRST, then the tool-specific improvements:

### Universal Improvements for Every Single Tool:

1. **Fallback Chain (Minimum 5-6 fallbacks):** Every tool that calls external APIs, processes files, or does computation must have a waterfall of fallback methods. If Method 1 fails, try Method 2 automatically and silently. Only show an error to the user if ALL methods fail.

2. **Input Validation:** Before processing, validate all inputs client-side. Show inline errors immediately without waiting for form submission.

3. **Loading State:** Show a spinner, skeleton screen, or progress bar during processing. Disable the action button during processing to prevent double-submits.

4. **Success Feedback:** After successful processing, give clear visual confirmation.

5. **Error Messages:** All error messages must be in plain English, friendly, and actionable. Never show raw error objects, stack traces, or HTTP status codes to users.

6. **Mobile Optimization:** Every tool must be fully usable on a small phone screen. Test layout on 360px wide viewport.

7. **Copy/Download Convenience:** Every text output needs a copy button. Every file output needs a download button with the correct filename and extension.

8. **Processing History (Nice to have):** Store the last 3-5 operations in localStorage so the user can recall their recent inputs without re-entering data.

9. **SEO Per Tool:** Every tool page needs a unique, descriptive title tag, meta description, and Open Graph tags.

10. **Performance:** Every tool page should load its tool-specific heavy libraries (like pdf.js, ffmpeg.wasm, tesseract.js) lazily — only when the user actually uses the tool. Do not load these in the global bundle.

### CATEGORY: VIDEO DOWNLOADER TOOLS

**CRITICAL SECTION — These are the most buggy tools. Requires the most fallbacks.**

**Tools:** YouTube Video Downloader, YouTube to MP3, YouTube to MP4, Instagram Video Downloader, Facebook Video Downloader, TikTok Downloader, Twitter/X Video Downloader, Vimeo Downloader, Reddit Video Downloader, Twitch Clip Downloader, Pinterest Video Downloader, SoundCloud Downloader, and 50+ more

**Root Cause of Failures:**
Video downloading tools fail because they rely on scraping video URLs from platforms that actively block, rate-limit, and change their internal URL structures. A single API or scraping method will fail frequently.

**Universal Fallback Architecture for ALL Video Downloaders:**

- **Fallback 1:** Primary method — direct URL extraction from the page source or public API endpoint.
- **Fallback 2:** Alternative extraction library (e.g., yt-dlp compatible service, alternative API).
- **Fallback 3:** Third-party free API (cobalt.tools API, y2mate API, etc.) — have 2-3 options.
- **Fallback 4:** Proxy-based extraction — route the request through a server proxy to bypass regional restrictions.
- **Fallback 5:** Alternative format/quality — if HD fails, try SD. If MP4 fails, try WebM.
- **Fallback 6:** Browser extension method — suggest the user a Chrome extension as last resort, but only if all else fails.

**All these fallbacks must be tried automatically and silently. The user should see at most a progress indicator, not the failure chain.**

**Improvements for YouTube Downloader specifically:**
- Show available quality options BEFORE user confirms download (720p, 480p, 360p, Audio only).
- Show video thumbnail, title, duration, and channel name after URL paste — confirmation before download.
- Format selector: MP4, WebM, MP3 (audio only).
- Progress indicator during download preparation and actual download.
- For long videos: chunked download with progress percentage.

**Improvements for Instagram Video Downloader:**
- Support Reels, Stories, IGTV, regular video posts.
- Show thumbnail + caption preview before download.
- Handle private accounts: show clear "This post is private — we cannot access it" message.

**Improvements for ALL Downloaders:**
- After a URL is pasted, show a preview (thumbnail, title if available) before the download button activates.
- Clear status messages: "Fetching video info... → Preparing download link... → Ready to download."
- If a URL is invalid or not from a supported platform, show immediate validation error.
- Mobile-friendly download: on iOS, which doesn't auto-download files, show instructions to long-press and save.

## PHASE 7 — ERROR HANDLING STANDARDS

This is the most important phase for reliability. Apply these to EVERY tool:

### Error Category 1: Network Errors
- Show a specific message: "Connection issue. Check your internet and try again."
- Auto-retry up to 3 times with exponential backoff before showing the error.
- Offer an offline alternative if one exists (client-side processing).

### Error Category 2: File Processing Errors
- Show which file caused the error and why (if determinable).
- Suggest fixes: "This PDF may be corrupt. Try opening it in a PDF viewer and re-saving."
- Never lose the user's uploaded file — keep it available for retry.

### Error Category 3: Input Validation Errors
- Show inline, adjacent to the input field.
- Never just highlight the field in red without explaining why.
- Clear the error as soon as the user corrects the input.

### Error Category 4: API Quota/Rate Limit Errors
- Switch to the next fallback API silently.
- If all APIs are exhausted, show: "Our service is busy right now. Please try again in a minute."
- Never expose API keys or API service names to the user.

### Error Category 5: Browser Compatibility Errors
- If a feature is not supported (e.g., SharedArrayBuffer for FFmpeg), show: "This tool requires a modern browser. Please update Chrome, Firefox, or Edge."
- Provide a server-side fallback whenever possible rather than just refusing to work.

- Prioritize fixing broken things over adding new features.
- The video downloader tools are the highest-priority fix. Get them working with robust fallbacks first.
- Document every change made in a CHANGELOG.md file in the repository.
