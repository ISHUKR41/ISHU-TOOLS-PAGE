## CRITICAL RULES BEFORE YOU BEGIN (READ FIRST, NEVER IGNORE)

1. **NO IMPACT ON TOOL FUNCTIONALITY** — UI/UX changes must NEVER break any existing tool logic. Test every tool after any frontend change. If a tool was working before, it must work after.
2. **NO CODE IS TO BE SHOWN TO THE OWNER** — The owner only wants working improvements delivered, not code explanations.
3. **MINIMUM 5-6 FALLBACKS PER TOOL** — Every single tool must have at least 5-6 fallback strategies. If Fallback 1 fails, Fallback 2 triggers automatically and silently. The user should never see a raw error.
4. **MULTIPLE LIBRARIES PER TOOL** — Never depend on a single library. Use primary + secondary + tertiary libraries wherever possible. Example: for video download, try yt-dlp → youtube-dl → cobalt API → rapidapi → custom scraper → direct fetch.
5. **DEBUG EVERYTHING FIRST** — Before improving any tool, run a full debug of all files and folders in the repo. Fix broken imports, dead code, undefined variables, missing error handlers, and silent failures first.
6. **PROFESSIONAL + SLIGHTLY MODERN UI** — Not too flashy, not old. Clean, fast, professional. Keep the existing ISHU TOOLS brand identity intact.
7. **ZERO WATERMARKS, ZERO SIGNUP WALLS** — This is a core promise of the platform. No improvement should add any friction.

## PHASE 0 — FULL REPOSITORY DEBUG (DO THIS FIRST)

Before touching any feature, perform a complete diagnostic of the GitHub repository at https://github.com/ISHUKR41/ISHU-TOOLS-PAGE.git

### Files & Folders to Debug
- Scan every single file in the repo for syntax errors, broken imports, missing exports, undefined references.
- Check all `package.json` / `requirements.txt` / dependency files for outdated, vulnerable, or missing packages.
- Identify all API endpoints and test each one — mark which are failing, slow, or returning wrong responses.
- Find all TODO/FIXME/HACK comments and resolve them.
- Look for hardcoded API keys or secrets exposed in client-side code — move all to environment variables.
- Identify all `console.error` and `throw` statements that are not caught — wrap them.
- Find all fetch/axios calls without timeout handling — add 10-second timeouts minimum.
- Check all file upload handlers — ensure they validate file type, size, and reject malicious inputs.
- Find any tool that relies on a single external API with no fallback — these are the highest-priority bugs.
- Scan for race conditions in tools that process multiple files simultaneously.
- Check that all output/download links expire properly and temp files are cleaned up.
- Verify all tools work on mobile (responsive CSS), on Chrome, Firefox, Safari, and Edge.
- Check all tools for accessibility basics: focus states, keyboard navigation, screen reader labels.
- Check loading states — every tool that does processing must show a spinner/progress indicator.

## PHASE 3 — DEBUGGING ALL FILES AND FOLDERS IN REPO

### Specific Debug Checklist

1. **Go through every route/page file** — check for missing error boundaries, unhandled promise rejections.
2. **Check all API handler files** — ensure every endpoint has: (a) input validation, (b) rate limiting, (c) file size limits, (d) timeout handling, (e) proper HTTP status codes.
3. **Check all utility/helper files** — look for silent return-null patterns where errors should be thrown.
4. **Check all file upload handlers** — validate MIME type (not just extension), validate actual file magic bytes, set max file size limits (50MB for video, 10MB for images, 20MB for PDF).
5. **Check environment variable usage** — no hardcoded keys. If a key is missing, tool should gracefully say "Service temporarily unavailable" not crash.
6. **Check memory leaks** — ensure all uploaded temp files are deleted after processing (use try/finally).
7. **Check concurrent request handling** — if 10 users upload a video simultaneously, the server should queue, not crash.
8. **Check all download link generation** — links should expire after 5 minutes and temp files cleaned.
9. **Check all loading states** — every tool that calls an API or runs heavy processing must have a spinner. No blank waiting.
10. **Check mobile responsiveness** — every tool page must be 100% usable on a 375px wide screen.
11. **Check all tool URLs** — no broken links, no 404 on tool pages, no redirect loops.
12. **Check SEO metadata** — every tool page must have a unique title and meta description.
13. **Check all external API integrations** — add try/catch around every external API call.
14. **Check for CORS issues** — file download endpoints must have proper CORS headers.
15. **Check all text encoding** — UTF-8 everywhere. No encoding errors for Hindi/Devanagari text input.
16. **Check all file output naming** — downloaded files should have meaningful names (not "output.pdf" but "ishutools-compressed-filename.pdf").
17. **Check all batch operation tools** — ensure ZIP creation doesn't fail for large batches.
18. **Check browser compatibility** — test on Chrome, Firefox, Safari, Edge. Fix any browser-specific issues.
19. **Check page load speed** — no tool page should take more than 2 seconds to become interactive. Lazy-load heavy libraries only when needed.
20. **Check Service Worker** — if already implemented, ensure it's not caching stale API responses.
